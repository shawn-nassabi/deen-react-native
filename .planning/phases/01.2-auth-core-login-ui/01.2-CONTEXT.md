# Phase 1.2: Auth Core + Login UI - Context

**Gathered:** 2026-04-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Rewrite the authentication layer to use Supabase email+password, and build the corresponding UI screens. Delivers:
- `hooks/useAuth.tsx` — AuthProvider rewired to `supabase.auth.onAuthStateChange`; public API preserved with updated signatures (MIG-03)
- `utils/auth.ts` — Cognito PKCE replaced with thin Supabase wrappers; `getValidAccessToken()` preserved (MIG-04, MIG-06)
- `app/login.tsx` — replaced with email+password form matching elevated card design (UI-01, UI-05, UI-06)
- `app/signup.tsx` — new Sign Up screen, same visual quality (AUTH-02, UI-02, UI-05, UI-06)
- Old `deen.auth.tokens` SecureStore key cleared on first auth restore (MIG-07)

Password reset, deep link handling, and Cognito package removal are out of scope (Phases 1.3 and 1.4).

</domain>

<decisions>
## Implementation Decisions

### Login / Sign Up UI Style
- **D-01:** Elevated card layout — logo sits above the card on the dark background (`colors.background`), card (`colors.panel`) contains all inputs, the primary button, and navigation links ('Forgot password?' + 'Don't have an account? Sign up').
- **D-02:** Bordered box input fields — each field has a visible rounded border using `colors.border`. Focused field gets an emerald (`#5bc1a1`) border highlight. Input background uses `colors.panel2` or `colors.panel` (Claude's discretion).
- **D-03:** 'Forgot password?' and 'Sign up' links are inside the card, below the button — not below the card as standalone text.
- **D-04:** Sign Up screen matches the Sign In screen visually: same card, same input style. Header text changes ('Create account' instead of 'Sign in to continue'), button text changes. No redesign needed.

### Screen Routing Structure
- **D-05:** Flat structure — `app/signup.tsx` alongside `app/login.tsx`. No new route group. Phase 1.3 will add reset-password screens at the same level.
- **D-06:** Navigation between auth screens uses `router.push('/signup')` from login, and `router.back()` (or `router.replace('/login')`) from signup to return. Standard Expo Router stack navigation.

### `useAuth` Hook API
- **D-07:** `signIn` signature changes to `signIn(email: string, password: string): Promise<void>`. Hook calls `supabase.auth.signInWithPassword()` and updates state. `login.tsx` passes credentials in; no Supabase import needed in the screen.
- **D-08:** `signUp(email: string, password: string): Promise<void>` is added to the hook. Symmetric with `signIn`. Sign Up screen uses `useAuth` the same way as Sign In.
- **D-09:** `AuthUser` type gains `id: string` (Supabase UUID). `sub?: string` is kept as an alias (`sub = id`) for backward compatibility with consumers that still use `user?.sub` — these are replaced in Phase 1.4. Shape: `{ id: string; sub?: string; email?: string }`.
- **D-10:** `useAuth` public API surface preserved: `status`, `user`, `accessToken`, `signIn`, `signOut`, `refresh` remain. Debug fields (`debug`, `lastAuthUrl`, `lastProxyStartUrl`, `lastReturnUrl`) are removed — they were Cognito-specific and have no Supabase equivalent.
- **D-11:** `AuthProvider` subscribes to `supabase.auth.onAuthStateChange` instead of manual token polling. Session state is read from the event callback's `session` argument — no manual `loadTokens()` or `getValidAccessToken()` call needed on startup.

### `utils/auth.ts` Restructuring
- **D-12:** Keep `utils/auth.ts` as a thin-wrapper module — gut all Cognito PKCE logic, replace with Supabase calls. `useAuth.tsx` continues to import from it. `api.ts` import is unchanged.
- **D-13:** `getValidAccessToken()` is preserved in `utils/auth.ts` as a thin wrapper: calls `supabase.auth.getSession()` and returns `session?.access_token ?? null`. `utils/api.ts` import stays the same — zero changes to `api.ts` for this requirement.
- **D-14:** All other Cognito-specific exports (`signInWithCognitoHostedUI`, `loadTokens`, `getAuthDebugInfo`, `getLastAuthorizeUrl`, etc.) are removed. `utils/auth.ts` exports only: `signIn`, `signOut`, `signUp` (thin wrappers), and `getValidAccessToken`.

### Old Token Cleanup (MIG-07)
- **D-15:** Clear `deen.auth.tokens` from `expo-secure-store` on first auth restore in `AuthProvider`. Detect by checking if the key exists; if so, delete it. No separate migration flag needed — the key is absent after first cleanup.

### Claude's Discretion
- Error message strings shown to the user for each failure state (wrong password, email already in use, network error, weak password) — pick clear, friendly messages consistent with the existing error text style.
- Exact card padding, border radius, and shadow values — use existing `styles` conventions from `app/login.tsx` as baseline, extend as needed.
- Whether the password field has a show/hide toggle — add if it feels natural, skip if out of scope for this phase.
- `AuthProvider` cleanup of the `onAuthStateChange` subscription — use the `data.subscription.unsubscribe()` return value in a `useEffect` cleanup function.

</decisions>

## Existing Code Insights

### Files Being Replaced / Modified

- `utils/auth.ts` — Full rewrite. Only the function signatures are preserved; all implementations change.
- `app/login.tsx` — Replace single-button layout with email+password card form. Keep `ThemedView`, `ThemedText`, `useAuth`, `useColorScheme`, `Colors` imports — they all still apply.
- `utils/api.ts` — `getValidAccessToken()` import from `./auth` stays unchanged (MIG-06 handled transparently by D-13).

### New Files
- `app/signup.tsx` — New screen. Mirror login.tsx structure and style; swap for `signUp()` call.

### Reusable Patterns
- `ThemedView` / `ThemedText` — use throughout auth screens (already in login.tsx).
- `Colors[colorScheme]` — all color references go through this; no hardcoded hex except universal `#fff` / `#000`.
- `StyleSheet.create({})` at bottom of file, named `styles`.
- Error state pattern from login.tsx: `const [error, setError] = React.useState<string | null>(null)` + inline `ThemedText` with `{ color: "#ef4444" }`.

### Consumers of `useAuth` (must continue working unchanged)
- `app/_layout.tsx` — reads `status` for auth-gated redirect
- `app/(tabs)/index.tsx`, `app/(tabs)/chat.tsx`, etc. — likely read `user` and `accessToken`
- `app/settings.tsx`, `app/feedback.tsx` — may read `user`
