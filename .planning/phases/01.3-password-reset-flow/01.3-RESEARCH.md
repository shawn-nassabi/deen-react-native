## Summary

Phase 1.3 delivers the full password-reset flow: a Forgot Password screen that sends a Supabase PKCE recovery email, and a New Password screen that exchanges the code from the deep link and saves the new password. Both screens mirror the elevated card layout established in Phase 1.2 (login.tsx, signup.tsx) with no new libraries required.

The Supabase SDK (v2.103.0 installed) provides all three API calls needed: `resetPasswordForEmail`, `exchangeCodeForSession`, and `updateUser`. The PKCE code verifier is stored automatically by the SDK in `LargeSecureStore` during `resetPasswordForEmail` and consumed automatically during `exchangeCodeForSession` â€” the implementer does not need to manage verifier storage manually.

After `updateUser` succeeds it fires a `USER_UPDATED` event with a valid session, which `useAuth.tsx`'s `onAuthStateChange` listener picks up (session truthy â†’ `setStatus("signedIn")`), triggering the `_layout.tsx` redirect to `/(tabs)` automatically as D-07 specifies. No manual redirect needed in reset-password.tsx.

**Primary recommendation:** Implement both screens directly as close copies of signup.tsx (which already has the same two-state card pattern: form view / confirmation view). The reset-password screen adds a third loading state before the form appears.

### Pattern 3: Code Exchange on Mount

```typescript
// Source: @supabase/auth-js GoTrueClient.ts, exchangeCodeForSession (line 1351)

import { useLocalSearchParams } from "expo-router";
import { supabase } from "@/utils/supabase";

const { code } = useLocalSearchParams<{ code?: string }>();

useEffect(() => {
  if (!code) {
    setState("error");
    return;
  }
  supabase.auth.exchangeCodeForSession(code)
    .then(({ error }) => {
      if (error) {
        setState("error");
      } else {
        setState("form");
      }
    });

```

### Anti-Patterns to Avoid

- **Passing `code` as a useEffect dependency:** The code param is stable for the component's lifetime. Adding it as a dependency would cause the exchange to re-run if Expo Router re-renders params (can cause `AuthPKCECodeVerifierMissingError` on second call because verifier is deleted after first exchange).
- **Calling `supabase.auth.exchangeCodeForSession` inside a `useCallback` without mount guard:** Add a mounted ref or empty deps array to prevent double invocation in React Strict Mode dev builds.
- **Calling `router.replace("/(tabs)")` after `updateUser` succeeds:** The `_layout.tsx` handles redirect via `onAuthStateChange`. A manual replace would race with the automatic one and may cause double navigation.
- **Using `useAuth().signIn` or any hook method for the code exchange:** These operations are direct `supabase` client calls, not part of the managed `useAuth` API. Import `supabase` directly from `@/utils/supabase`.
- **Missing `code` param guard:** The deep link `deenreactnative://reset-password` without a query string is a valid URL that Expo Router will route to `reset-password.tsx`. The `code` param will be `undefined`. Always guard: `if (!code) { setState("error"); return; }`.

## Common Pitfalls

### Pitfall 1: AuthPKCECodeVerifierMissingError on Exchange

**What goes wrong:** `exchangeCodeForSession(code)` throws `AuthPKCECodeVerifierMissingError` even though the code is valid.

**Why it happens:** The PKCE code verifier is stored in `LargeSecureStore` during `resetPasswordForEmail`. If the verifier key does not exist when `exchangeCodeForSession` runs, the SDK throws this error. This happens if: (a) the app was uninstalled and reinstalled between requesting the reset email and tapping the link (SecureStore cleared), or (b) the effect runs twice (React Strict Mode double-invoke) and the verifier was deleted on the first run.

**How to avoid:** This is a legitimate user-error state (uninstall/reinstall, long delay). Handle it the same as any other `exchangeCodeForSession` error â€” show the "Link expired" state with "Request a new one" prompt. For Strict Mode double-invoke, use a mount guard:
```typescript
const exchanged = React.useRef(false);
useEffect(() => {
  if (exchanged.current) return;
  exchanged.current = true;
  // ... exchangeCodeForSession call
}, []);
```

**Warning signs:** Error message contains "code_verifier" or "PKCE".

### Pitfall 2: Deep Link Not Opening App on Expo Go

**What goes wrong:** Tapping the reset link opens a browser instead of the app, or the app opens but the `code` param is missing.

**Why it happens:** Custom scheme deep links (`deenreactnative://`) require a standalone or dev-client build. Expo Go intercepts custom scheme links differently.

**How to avoid:** Phase 1.3 must be tested on a dev-client build only. Document this clearly in the plan. This is a known constraint from STATE.md and CONTEXT.md.

**Warning signs:** Testing on Expo Go scan; link opens browser or Expo Go home.

### Pitfall 3: isOnAuthScreen Missing reset-password â†’ Redirect Loop

**What goes wrong:** User taps reset link, app opens at `reset-password`, but `_layout.tsx` immediately redirects to `/login` because `status === "signedOut"` and `isOnAuthScreen` is `false`.

**Why it happens:** `_layout.tsx` lines 46â€“50 â€” if `isOnAuthScreen` does not include `"reset-password"`, the unauthenticated user is redirected away before the code exchange can happen.

**How to avoid:** D-10 extension of `isOnAuthScreen` is a mandatory change. It must happen in the same plan/task as or before the new screens are created.

**Warning signs:** Deep link redirects to login instead of showing spinner.

### Pitfall 4: Stack.Screen Entries Missing â†’ Expo Router Warning

**What goes wrong:** Navigating to `/forgot-password` or `/reset-password` works but Expo Router logs a warning about an unregistered screen.

**Why it happens:** Expo Router requires `<Stack.Screen>` entries for screens that should not show the default header.

**How to avoid:** D-11 â€” add both entries to `<Stack>` in `_layout.tsx`.

### Pitfall 5: USER_UPDATED vs SIGNED_IN Event Confusion

**What goes wrong:** Developer assumes `updateUser` fires `SIGNED_IN` and adds manual state manipulation, creating double-state-update bugs.

**Why it happens:** `updateUser` fires `USER_UPDATED` (confirmed: GoTrueClient.ts line 3149 `_notifyAllSubscribers('USER_UPDATED', session)`). The `useAuth.tsx` listener handles this correctly because it checks `if (session)` regardless of event type â€” but adding manual state manipulation would conflict.

**How to avoid:** Do not manually call `setStatus`, `setUser`, or `router.replace` after `updateUser` succeeds. Trust the existing listener chain.

## Open Questions

1. **Supabase Dashboard PKCE Setting**
   - What we know: CONTEXT.md D-05 states "Supabase dashboard 'Use PKCE flow' is ON" as a confirmed decision.
   - What's unclear: The plan must include a verification note that the dashboard has PKCE enabled. If it is not enabled, `resetPasswordForEmail` will send a `#access_token=` hash-fragment link instead of a `?code=` query param, and `exchangeCodeForSession` will fail with no code.
   - Recommendation: Add a "verify Supabase dashboard PKCE setting is ON" step in Wave 0 or as a pre-condition note on the reset-password task. This was also flagged in STATE.md Blockers.

2. **React Strict Mode double-invoke**
   - What we know: New Architecture is enabled (`newArchEnabled: true`). React Strict Mode may double-invoke `useEffect` in development.
   - What's unclear: Whether the mount guard `useRef(false)` is sufficient or whether Expo's dev build configuration disables Strict Mode effects double-invoke.
   - Recommendation: Use the mount guard pattern shown in Code Examples. It is safe in both Strict Mode and production.
