## Summary

This phase installs and wires the Supabase JS SDK as a pure infrastructure step — no auth UI, no hook changes. Three files change: `utils/polyfills.ts` gains the URL polyfill import, `utils/supabase.ts` is created from scratch, and `utils/config.ts` has Cognito fields stripped and Supabase fields added.

The `@supabase/supabase-js` package is not yet installed (confirmed). Its latest stable release is **2.103.0** (published 2026-04-09), which is far above the v2.50+ threshold where `startAutoRefresh` / `stopAutoRefresh` were introduced — the AppState wiring can proceed without the fallback path described in D-04.

**Critical clarification on LargeSecureStore approach:** D-01 (use official guide verbatim) and D-02 (numeric suffix chunking) describe two different approaches. The official Supabase React Native guide's LargeSecureStore uses **AES-256 encryption** — it stores an encryption key in SecureStore and the encrypted session payload in AsyncStorage. It does NOT use raw byte-chunking with `.0` / `.1` / `.count` suffix keys. D-02's description matches older community patterns or the SSR cookie-chunking mechanism, not the official React Native guide. The planner must choose: follow D-01 (official guide = AES approach) or D-02 (numeric suffix chunking). Research recommends D-01 (AES approach) because it is the officially documented pattern and avoids implementing custom chunking logic. Flag this to the executor.

**Primary recommendation:** Follow D-01 (official guide verbatim = AES-256 encrypted LargeSecureStore). Install `@supabase/supabase-js@latest` and `react-native-url-polyfill`. The official guide also requires `aes-js` and `react-native-get-random-values`. If D-02's explicit numeric-suffix chunking is required instead, that is a fully custom implementation with no official reference — the planner should surface this choice to the user before execution.

### Pattern 1: URL Polyfill First Import

**What:** `react-native-url-polyfill/auto` must be the first import in `utils/polyfills.ts`. The polyfill patches the `URL` and `URLSearchParams` globals so Hermes has them when Supabase's internal code runs.

**When to use:** Required because Hermes (React Native JS engine) ships without a spec-compliant `URL` implementation. Supabase `createClient` calls `new URL(...)` at module initialisation.

**Example:**
```typescript
// utils/polyfills.ts
/**

import "react-native-url-polyfill/auto";

```

### Pattern 2: LargeSecureStore — AES-256 Encryption Approach (Official Guide / D-01 path)

**What:** Stores an AES-256 key in SecureStore (under the session key name) and the encrypted session JSON in AsyncStorage (under the same key). SecureStore holds only the 32-byte hex key, well under the 2048-byte limit.

**When to use:** When following the official Supabase React Native guide verbatim (D-01).

**Example:**
```typescript
// Source: https://supabase.com/docs/guides/getting-started/tutorials/with-expo-react-native?auth-store=secure-store
import * as aesjs from "aes-js";
import * as SecureStore from "expo-secure-store";
import AsyncStorage from "@react-native-async-storage/async-storage";
import { type SupportedStorage } from "@supabase/supabase-js";

class LargeSecureStore implements SupportedStorage {
  private async _encrypt(key: string, value: string): Promise<string> {
    const encryptionKey = crypto.getRandomValues(new Uint8Array(256 / 8));
    const cipher = new aesjs.ModeOfOperation.ctr(
      encryptionKey,
      new aesjs.Counter(1)
    );
    const encryptedBytes = cipher.encrypt(aesjs.utils.utf8.toBytes(value));
    await SecureStore.setItemAsync(
      key,
      aesjs.utils.hex.fromBytes(encryptionKey)
    );
    return aesjs.utils.hex.fromBytes(encryptedBytes);
  }

  private async _decrypt(key: string, value: string): Promise<string | null> {
    const encryptionKeyHex = await SecureStore.getItemAsync(key);
    if (!encryptionKeyHex) return null;
    const cipher = new aesjs.ModeOfOperation.ctr(
      aesjs.utils.hex.toBytes(encryptionKeyHex),
      new aesjs.Counter(1)
    );
    const decryptedBytes = cipher.decrypt(aesjs.utils.hex.toBytes(value));
    return aesjs.utils.utf8.fromBytes(decryptedBytes);
  }

  async getItem(key: string): Promise<string | null> {
    try {
      const encrypted = await AsyncStorage.getItem(key);
      if (!encrypted) return null;
      return await this._decrypt(key, encrypted);
    } catch (e) {
      console.warn("LargeSecureStore.getItem failed:", e);
      return null;
    }
  }

  async removeItem(key: string): Promise<void> {
    try {
      await AsyncStorage.removeItem(key);
      await SecureStore.deleteItemAsync(key);
    } catch (e) {
      console.warn("LargeSecureStore.removeItem failed:", e);
    }
  }

  async setItem(key: string, value: string): Promise<void> {
    try {
      const encrypted = await this._encrypt(key, value);
      await AsyncStorage.setItem(key, encrypted);
    } catch (e) {
      console.warn("LargeSecureStore.setItem failed:", e);
    }
  }
}
```

### Pattern 3: LargeSecureStore — Numeric-Suffix Chunking Approach (D-02 path)

**What:** Splits the session string into 2048-byte chunks and stores each under `<key>.0`, `<key>.1`, ..., `<key>.count`. No encryption dependencies needed. Uses only `expo-secure-store` (already installed).

**When to use:** If D-02 is confirmed as the chosen approach instead of the official guide (D-01). This is a DIY implementation — no official Supabase reference.

**Example (community pattern — LOW confidence, no official source):**
```typescript
// Community pattern — NOT from official Supabase docs
import * as SecureStore from "expo-secure-store";
import { type SupportedStorage } from "@supabase/supabase-js";

const CHUNK_SIZE = 2048;

class LargeSecureStore implements SupportedStorage {
  async getItem(key: string): Promise<string | null> {
    try {
      const countStr = await SecureStore.getItemAsync(`${key}.count`);
      if (!countStr) return null;
      const count = parseInt(countStr, 10);
      const chunks: string[] = [];
      for (let i = 0; i < count; i++) {
        const chunk = await SecureStore.getItemAsync(`${key}.${i}`);
        if (chunk === null) return null;
        chunks.push(chunk);
      }
      return chunks.join("");
    } catch (e) {
      console.warn("LargeSecureStore.getItem failed:", e);
      return null;
    }
  }

  async removeItem(key: string): Promise<void> {
    try {
      const countStr = await SecureStore.getItemAsync(`${key}.count`);
      if (countStr) {
        const count = parseInt(countStr, 10);
        for (let i = 0; i < count; i++) {
          await SecureStore.deleteItemAsync(`${key}.${i}`);
        }
      }
      await SecureStore.deleteItemAsync(`${key}.count`);
    } catch (e) {
      console.warn("LargeSecureStore.removeItem failed:", e);
    }
  }

  async setItem(key: string, value: string): Promise<void> {
    try {
      const chunks: string[] = [];
      for (let i = 0; i < value.length; i += CHUNK_SIZE) {
        chunks.push(value.slice(i, i + CHUNK_SIZE));
      }
      for (let i = 0; i < chunks.length; i++) {
        await SecureStore.setItemAsync(`${key}.${i}`, chunks[i]);
      }
      await SecureStore.setItemAsync(`${key}.count`, String(chunks.length));
    } catch (e) {
      console.warn("LargeSecureStore.setItem failed:", e);
    }
  }
}
```

### Pattern 4: Supabase Singleton + AppState Wiring

**What:** Single exported `supabase` constant created once at module load. `AppState` listener starts/stops token auto-refresh based on foreground/background state.

**When to use:** Always — this is the required pattern for non-browser platforms.

**Example (verified via multiple Supabase community sources):**
```typescript
// Source: Official Supabase React Native quickstart + ignitecookbook.com pattern
import "react-native-url-polyfill/auto";
import { createClient } from "@supabase/supabase-js";
import { AppState } from "react-native";
import { CONFIG } from "@/utils/config";

// LargeSecureStore defined above ...

export const supabase = createClient(
  CONFIG.SUPABASE_URL,
  CONFIG.SUPABASE_PUBLISHABLE_KEY,
  {
    auth: {
      storage: new LargeSecureStore(),
      autoRefreshToken: true,
      persistSession: true,
      detectSessionInUrl: false,
    },
  }
);

// Tells Supabase Auth to continuously refresh the session automatically
// if the app is in the foreground. When this is added, you will continue
// to receive `onAuthStateChange` events with the `TOKEN_REFRESHED` event
// if the session is set to expire while the app is in the foreground.
// When the app moves to the background, you should stop the auto refresh to
// avoid consuming network resources unnecessarily.
AppState.addEventListener("change", (state) => {
  if (state === "active") {
    supabase.auth.startAutoRefresh();
  } else {
    supabase.auth.stopAutoRefresh();
  }
});
```

> **Alternative fail-fast pattern** (simpler, TypeScript-friendly): use a guard function at module top that throws before `CONFIG` is even built. Both produce a recognizable error; the inline throw is more common in the codebase's style.

### Anti-Patterns to Avoid

- **Importing `supabase` in `utils/polyfills.ts`:** The polyfill file is imported first in `_layout.tsx` — if it imports `supabase`, the client initializes before polyfills run, causing the `URL is not defined` crash.
- **Hardcoding Supabase URL or key:** D-06 prohibits fallback values. A missing env var should fail loudly at startup.
- **Adding AppState listener in a hook or component:** D-05 co-locates it in `utils/supabase.ts`. Putting it in a hook means multiple registrations if the hook mounts more than once.
- **Placing `import 'react-native-url-polyfill/auto'` inside `utils/supabase.ts`:** The polyfill must run before any Supabase import. The canonical way is `utils/polyfills.ts` (already imported first in `_layout.tsx`). Putting it inside `supabase.ts` risks import-order issues if `supabase.ts` is ever imported before `_layout.tsx` processes polyfills.
- **Using `as const` on CONFIG with the throw pattern:** TypeScript cannot infer the exact type of a throwing expression. Use a pre-validated string cast: validate env vars early, then assert as string.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Token auto-refresh scheduling | Custom `setInterval` / expiry check | `autoRefreshToken: true` + AppState listener | Supabase SDK handles refresh timing, jitter, retry backoff |
| Session persistence serialization | Custom JSON encode/store | Supabase SDK + LargeSecureStore adapter | SDK controls the storage key names and serialization format |
| Token expiry detection | Manual `exp` field parsing + clock skew | `supabase.auth.getSession()` | SDK always returns a valid (auto-refreshed) session or null |
| URL polyfill bootstrapping | Custom URL shim | `react-native-url-polyfill/auto` | Battle-tested; handles edge cases in URL parsing that a hand-rolled shim won't |

**Key insight:** The Supabase SDK is designed to own the full auth session lifecycle. Fighting it with custom refresh logic is the most common cause of "session disappears" bugs.

## Common Pitfalls

### Pitfall 1: `ReferenceError: URL is not defined` on first import

**What goes wrong:** App crashes with white screen immediately on launch. Error appears in Metro logs before any screen renders.

**Why it happens:** Supabase `createClient` calls `new URL(...)` at module evaluation time. If `react-native-url-polyfill/auto` hasn't run yet, `URL` is undefined on Hermes.

**How to avoid:** `react-native-url-polyfill/auto` must be the first import in `utils/polyfills.ts`. Verify `_layout.tsx` imports polyfills as its very first line (it already does: `import "@/utils/polyfills";`).

**Warning signs:** White screen on launch, Metro logs showing `ReferenceError: URL is not defined` or `Cannot read property of undefined (reading 'protocol')`.

### Pitfall 2: `startAutoRefresh is not a function`

**What goes wrong:** AppState listener throws at runtime when app comes back to foreground.

**Why it happens:** `startAutoRefresh` and `stopAutoRefresh` were introduced in `@supabase/auth-js` v2.50 (surfaced as `@supabase/supabase-js` v2.50+). An older installed version won't have these methods.

**How to avoid:** With `@supabase/supabase-js@2.103.0` (the version that will be installed), these methods definitively exist. Still verify after install: `npm list @supabase/supabase-js` should show 2.50+. D-04's fallback path (omit AppState wiring) is not needed for 2.103.0.

**Warning signs:** `TypeError: supabase.auth.startAutoRefresh is not a function` in logs.

### Pitfall 3: `expo-secure-store` 2048-byte limit

**What goes wrong:** `setItemAsync` throws `SecureStore.setItemAsync: Value cannot be larger than 2048 bytes.` when Supabase tries to save a session.

**Why it happens:** Supabase sessions are typically 2–4 KB. Writing directly to SecureStore without chunking or encryption indirection fails.

**How to avoid:** LargeSecureStore is the solution. Both the AES approach (D-01) and chunking approach (D-02) solve this. Choosing neither and using a plain `SecureStore` adapter is the pitfall.

**Warning signs:** `Error: SecureStore.setItemAsync: Value cannot be larger than 2048 bytes` on first sign-in.

### Pitfall 4: Cognito fields still referenced after `utils/config.ts` update

**What goes wrong:** `utils/auth.ts` still imports `CONFIG.COGNITO_DOMAIN`, `CONFIG.COGNITO_CLIENT_ID`, etc. After removing those fields, TypeScript compilation fails.

**Why it happens:** Phase 1.1 removes Cognito fields from `CONFIG` but `utils/auth.ts` still references them (it is unchanged in Phase 1.1).

**How to avoid:** `utils/auth.ts` is NOT changed in Phase 1.1. The Cognito fields in `utils/config.ts` CANNOT be removed yet — they are still consumed by `utils/auth.ts` which remains active through Phase 1.2. **The MIG-10 removal of Cognito fields from `utils/config.ts` must be deferred to Phase 1.2** when `utils/auth.ts` is replaced. Phase 1.1 only ADDS the Supabase fields to `CONFIG`; it does not remove Cognito fields. This is a critical sequencing constraint.

**Warning signs:** TypeScript errors on `CONFIG.COGNITO_DOMAIN` (or similar) in `utils/auth.ts` after removing the fields.

### Pitfall 5: Double AppState listener registration

**What goes wrong:** Token refresh fires multiple times per foreground event; auth state flickers.

**Why it happens:** `AppState.addEventListener` is called every time the module is evaluated. In React Native, modules are cached after first load, so this normally runs once — but if the file is hot-reloaded in development, the listener can stack.

**How to avoid:** Module-level registration (not inside a function call) is safe for production. In development, if flickering is observed, use `AppState.addEventListener` return value to remove the old listener before adding a new one.

### Pitfall 6: `CONFIG` `as const` incompatible with throwing initializer

**What goes wrong:** TypeScript reports type error when mixing `() => { throw ... }` with `as const`.

**Why it happens:** `as const` requires literal types. A function call expression does not have a literal type.

**How to avoid:** Validate env vars before building `CONFIG` using a guard at module top:

```typescript
// Validate at module load — fails loudly with a recognizable message
function requireEnv(name: string): string {
  const value = process.env[name];
  if (!value) throw new Error(`Missing required env var: ${name}`);
  return value;
}

const SUPABASE_URL = requireEnv("EXPO_PUBLIC_SUPABASE_URL");
const SUPABASE_PUBLISHABLE_KEY = requireEnv("EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY");

export const CONFIG = {
  // ...existing fields...
  SUPABASE_URL,
  SUPABASE_PUBLISHABLE_KEY,
} as const;
```
