<decisions>
## Implementation Decisions

### LargeSecureStore Adapter
- **D-01:** Use the official Supabase React Native guide pattern verbatim — no community library, no extra dependency.
- **D-02:** Chunked into 2048-byte pieces using numeric suffix keys (e.g. `<key>.0`, `<key>.1`, `<key>.count`). Key names are driven by whatever the Supabase client passes in — no project-level prefix override needed.

### Supabase Client Options
- **D-03:** Full client config in `utils/supabase.ts`: `detectSessionInUrl: false`, `autoRefreshToken: true`, `persistSession: true`, and AppState listener wiring (`startAutoRefresh` / `stopAutoRefresh`).
- **D-04:** Install `@supabase/supabase-js@latest`. Before wiring the AppState listener, the planner/executor must verify that `startAutoRefresh` and `stopAutoRefresh` exist on the installed version (introduced in v2.50+). If missing, fall back to `autoRefreshToken: true` only and flag for Phase 1.2.
- **D-05:** AppState listener lives in `utils/supabase.ts` alongside the singleton — not in a hook or screen. This keeps the lifecycle management co-located with the client and avoids touching `hooks/useAuth.tsx` before Phase 1.2.

### Config & Key Naming
- **D-06:** Strict env-vars only — no hardcoded fallback values for Supabase URL or key. If env vars are missing, the app will fail with a recognizable error rather than silently using stale/wrong values.
- **D-07:** Env var name is `EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY` (not `EXPO_PUBLIC_SUPABASE_ANON_KEY`). The publishable key (`sb_publishable_...`) is the modern Supabase key format — easier to rotate, not tied to the JWT secret. The legacy anon key still works but is considered legacy.
- **D-08:** `EXPO_PUBLIC_COGNITO_DOMAIN`, `EXPO_PUBLIC_COGNITO_CLIENT_ID`, `EXPO_PUBLIC_COGNITO_ISSUER`, `EXPO_PUBLIC_COGNITO_SCOPES`, and `EXPO_PUBLIC_AUTH_REDIRECT_URI` are removed from `utils/config.ts`. The `CONFIG` object loses all Cognito fields and gains `SUPABASE_URL` and `SUPABASE_PUBLISHABLE_KEY`.

### Claude's Discretion
- TypeScript types for the `LargeSecureStore` class — implement as a class satisfying the `SupportedStorage` interface from `@supabase/supabase-js`.
- Error handling within `LargeSecureStore.getItem` / `setItem` / `removeItem` — follow project convention: `try/catch`, log with `console.warn`, return `null` on failure.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `expo-secure-store` v15 is already installed and used for `deen.auth.tokens` — LargeSecureStore reuses this same package, no new installation needed.
- `utils/polyfills.ts` is effectively empty (one comment, no imports) — adding `react-native-url-polyfill/auto` as the first import is clean and non-breaking.
- `utils/config.ts` — exports a `CONFIG` const object; the shape changes (Cognito fields out, Supabase fields in) but the export pattern stays the same.

### Established Patterns
- Error handling: `try/catch` in async storage ops, log with `console.warn`, return `null` / safe fallback.
- Singleton pattern for `utils/`: one default export per utility module (e.g., `api.ts` exports functions; `auth.ts` exports functions). `utils/supabase.ts` will export the singleton client as a named export: `export const supabase = createClient(...)`.
- File-level JSDoc block at top of each utility file describing purpose.

### Integration Points
- `app/_layout.tsx` imports `utils/polyfills.ts` as its first import — MIG-01's polyfill addition slots in naturally here.
- `utils/api.ts` currently calls `getValidAccessToken()` from `utils/auth.ts` — Phase 1.2 will update this to use `supabase.auth.getSession()`. Phase 1.1 only creates the client; no API wiring needed yet.
- `hooks/useAuth.tsx` is untouched in Phase 1.1 — the client singleton is created but not yet consumed by the auth hook.
